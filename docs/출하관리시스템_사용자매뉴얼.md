# 1. 시스템 개요

## 1.1 시스템 소개

PTLPSM(출하관리 시스템)은 공공조달 물류/출하 업무를 통합 관리하는 시스템입니다. 발주부터 납품완료, 자금 정산까지 전체 프로세스를 지원합니다.

- **관리자 웹**(PC): 발주·출하·납품·자금 등 모든 관리 업무. 주소 체계는 `/admin/...`
- **모바일 납품확인**(스마트폰): 현장 인수자/서명자가 로그인 없이 URL로 접속. 주소 체계는 `/m/delivery/{토큰}`

> 📷 [스크린샷: 로그인 화면 및 관리자 메인 대시보드 — 좌측 메뉴 + 상단 사용자/알림]

### 주요 특징

| 기능 | 설명 |
|------|------|
| **PDF 자동 추출** | 납품요구서 PDF를 업로드하면 AI/OCR로 정보 자동 추출 |
| **모바일 납품확인** | 현장에서 스마트폰으로 서명, 사진 촬영, GPS 위치 기록 |
| **다중 서명** | 현장소장 + 감리원 서명을 통한 승인 체계 |
| **자금 추적** | 선급금 → 기성금 → 잔금 → OEM 지급 전체 관리 |
| **문서 자동 생성** | 납품확인서, 납품완료계, 선급금 문서 등 자동 생성 |

## 1.2 전체 업무 흐름

### 기본 흐름도

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  발주   │ →  │  출하    │ →  │  운송    │ →  │  납품    │ →  │납품완료  │ →  │자금관리  │
│  등록   │    │  등록    │    │  배차    │    │  확인    │    │   계    │    │  정산   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
     │              │              │              │              │              │
     ▼              ▼              ▼              ▼              ▼              ▼
 납품요구서       출하지시서      운송장          모바일서명      3종PDF        선급/기성
 PDF 추출        발행           발행           사진촬영        생성          잔금/OEM
```

### 데이터 관계도

```
발주(Order) ──── 1:N ────→ 출하(Shipment) ──── 1:1 ────→ 운송(Transport)
    │                          │                              │
    │                          └──────────── 1:1 ─────────────┤
    │                                                         ▼
    │                                                  납품확인(Delivery)
    │                                                         │
    │                                                         ▼
    │                                                  인수증 PDF 생성
    │
    ├──── 1:N ────→ 기성차수(Baseline) ──→ 납품확인서/사진대지 PDF
    │
    ├──── 1:1 ────→ 납품완료계(DeliveryDone) ──→ 3종 PDF
    │
    └──── 1:1 ────→ 자금(Fund)
                      ├── 선급금 (1회)
                      ├── 기성금 (N회) ←── 기성차수와 연결
                      └── 잔금 (1회)
```

### 핵심 관계 설명

| 관계 | 비율 | 의미 |
|------|------|------|
| 발주 → 출하 | **1:N** | 발주 1건에 여러 번 분할 출하 가능 |
| 출하 → 운송 | **1:1** | 출하 1건당 운송장 1건 |
| 운송 → 납품확인 | **1:1** | 운송 완료 시 납품확인 1건 |
| 발주 → 기성차수 | **1:N** | 기성금 분할 청구 가능 |
| 발주 → 자금 | **1:1** | 발주별 자금 관리 |

## 1.3 사용자 역할

권한은 역할 코드(roleCode) 기준으로 부여됩니다. **시스템관리자(SYSTEM_ADMIN)** 와 **리드파워담당자(LEADPOWER_MANAGER)** 는 전체 기능에 접근할 수 있고(전체 접근 권한), 그 외 역할은 담당 업무로 제한됩니다. 시스템관리 메뉴(시스템 설정·접근로그·방문통계 등)는 **SYSTEM_ADMIN 전용**입니다.

| 역할 | 코드 | 주요 업무 | 접근 범위 |
|------|------|----------|----------|
| 시스템 관리자 | SYSTEM_ADMIN | 사용자/권한 관리, 시스템 설정, 통계 | 전체(시스템관리 포함) |
| 리드파워 담당자 | LEADPOWER_MANAGER | 발주/출하/납품/자금 관리 | 전체(시스템관리 제외) |
| 영업 담당자 | SALES | 수주/견적/영업 관리 | 담당 업무 |
| 현장 소장 | SITE_MANAGER | 납품확인·기성 서명 | 모바일 서명 |
| 현장 감리원 | SITE_INSPECTOR | 납품확인 검수·서명 | 모바일 서명 |
| 운송 담당자 | DRIVER | 배송 상태 업데이트 | 운송 |

> 메뉴별 노출/접근 권한은 **기초정보 → 메뉴권한(`/admin/basic-info/menu-auth`)** 에서 역할별로 관리합니다. (역할 코드 명칭이 코드와 다를 수 있어 일부는 `(확인 필요)`)

## 1.4 용어 정의

### 1.4.1 핵심 엔티티(업무 단위)

| 용어 | 영문/테이블 | 설명 | 다른 표현 |
|------|------------|------|-----------|
| 발주 | Order / orders | 공공조달 계약에 따른 물품 납품 요청 단위 | 납품요구·수주·계약 |
| 출하 | Shipment / shipments | 창고에서 현장으로 물품을 보내는 단위 | 배송·출고 |
| 운송 | Transport / transports | 출하 물품의 운송 정보(차량·기사) | 배차·운송장 |
| 납품확인 | Delivery / deliveries | 현장에서 물품 인수를 확인하는 절차 | 인수·수령확인 |
| 납품사진 | DeliveryPhoto | 납품 현장 증빙 사진 | - |
| 기성차수 | Baseline / baselines | 기성금 청구를 위한 납품 완료 단위(1·2·3차…) | 기성·차수 |
| 기성차수품목 | BaselineItem | 기성 차수의 품목별 수량/금액 스냅샷 | - |
| 자금 | Fund / funds | 발주별 자금 관리 정보 | 자금관리 |
| 선급금 | AdvancePayment | 계약 체결 후 선지급 금액(통상 70%) | - |
| 기성금 | ProgressPayment | 납품확인 완료 물량에 대한 대금 | 기성청구 |
| 잔금 | BalancePayment | 납품완료 후 정산되는 나머지(=계약금액−선급금−기성금누계) | - |
| OEM지급 | OemPayment | 협력업체 지급액(=기성금×70%) | - |
| 납품완료계 | DeliveryDone / delivery_done | 전체 납품 완료 후 조달청 제출용 보고서 | 납품완료·준공 |
| 품목 | Item / items | 납품 대상 물품 정보 | 물품·제품 |
| 회사 | Company / companies | 건설사·제조사 등 협력업체 정보 | - |

### 1.4.2 자주 쓰는 용어

| 용어 | 설명 |
|------|------|
| 납품요구번호 | 조달청 발행 고유 번호(예: `R26TB0...-00`). `-00`=본계약, `-01` 이상=변경계약 |
| 본계약 / 변경계약 / 추가계약 | 최초 계약 / 기존 계약 내용 변경 / 별도 추가 계약 |
| 품대계(품목총액) | 품목 금액 합계. **매출·집계·정산의 단일 기준 금액** |
| 발주수량/출하수량/납품수량/잔여수량 | 계약 수량 / 출고한 수량 / 납품확인된 수량 / 남은 수량 |
| 선급금비율 | 계약금액 대비 선급금 비율(기본 70%) |
| 사진대지 | 납품 현장 사진을 모은 증빙 문서(출하당 1~2장) |

## 1.5 업무 프로세스 상세 흐름

### 1.5.1 발주 → 출하 → 납품확인 사이클

발주 1건에 대해 **계약 물량을 모두 채울 때까지** 출하를 반복합니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                           발주 등록                                  │
│              (납품요구번호, 품목, 수량, 금액 등록)                    │
│                    예: 품목A 100개, 품목B 50개                       │
└────────────────────────────┬────────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                        출하 등록 (1차)                               │
│                   품목A: 100개 중 30개 출하                          │
│                   품목B: 50개 중 20개 출하                           │
└────────────────────────────┬────────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                        운송 배차 (1:1)                               │
│               차량번호, 기사명, 배송 예정일 등록                      │
└────────────────────────────┬────────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                     모바일 납품확인 (1:1)                            │
│                  인수자 서명 + 사진촬영 + GPS                        │
└────────────────────────────┬────────────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      인수증 PDF 자동 생성                            │
│                   (출하 완료, 기성금 청구 가능)                       │
└────────────────────────────┬────────────────────────────────────────┘
                             ▼
              ┌──────────────┴──────────────┐
              │     남은 물량이 있는가?      │
              └──────────────┬──────────────┘
                    예 │           │ 아니오
                       ▼           ▼
┌──────────────────────────┐  ┌────────────────────────────────────┐
│     출하 등록 (2차)       │  │       모든 물량 납품확인 완료        │
│  품목A: 남은 70개 중 40개 │  │    → 납품완료계 처리 가능 상태      │
│  품목B: 남은 30개 중 30개 │  └────────────────────────────────────┘
└──────────────────────────┘
         │
         ▼
    (반복 진행)
```

### 1.5.2 기성금 청구 프로세스

**납품확인이 완료된 출하**만 기성금 청구에 포함할 수 있습니다.

```
납품확인 완료된 출하 확인 (인수증 서명 완료분만)
        ▼
기성 차수 생성 (청구할 출하 선택 → 품목별 수량/금액 스냅샷)
        ▼
서명 URL 발송 (현장소장 + 감리원에게 LMS/SMS)
        ▼
현장소장 서명(PARTIAL_SIGNED) → 감리원 서명(SIGNATURE_COMPLETED)
        ▼
납품확인서 + 사진대지 PDF 자동 생성 (차수 상태: CONFIRMED)
        ▼
기성금 수금 확인 (입금일·입금금액 확인 후 PAID)
```

### 1.5.3 선급금 vs 기성금 vs 잔금

| 구분 | 신청 시점 | 신청 조건 | 금액 기준 | 서명 필요 |
|------|----------|----------|----------|----------|
| **선급금** | 첫 출하 전후(계약 초기) | 출하 여부 무관, **1회만** | 계약금액 × 비율(70%) | X |
| **기성금** | 납품 진행 중(여러 차수) | 납품확인 완료된 출하 | 실제 납품수량 × 단가 | O (현장소장+감리원) |
| **잔금** | 납품완료 후 | 납품완료계 완료 | 계약금액 − 선급금 − 기성금누계 | X |

**자금 흐름 예시:**
```
계약금액: 1억원
    ├─→ 선급금: 7천만원 (70%)        ←── 첫 출하 전 신청
    ├─→ 1차 기성금: 1천만원           ←── 1·2차 출하 납품확인 완료 후
    ├─→ 2차 기성금: 1천만원           ←── 3·4차 출하 납품확인 완료 후
    └─→ 잔금: 1천만원 (1억−7천−1천−1천) ←── 납품완료계 서명 완료 후
```

### 1.5.4 납품완료 시 생성 서류

| 서류 | 내용 | 용도 | 생성 시점 |
|------|------|------|----------|
| 인수증 | 출하별 납품 내역 + 인수자 서명 | 개별 납품 증빙 | 납품확인 완료 시 |
| 납품확인서 | 기성 차수별 품목 내역 + 양측 서명 | 기성금 청구 증빙 | 기성 차수 서명 완료 시 |
| 사진대지 | 현장 사진 모음(출하당 1~2장) | 납품 현장 증빙 | 기성 차수 서명 완료 시 |
| 납품완료계 | 전체 납품 요약 보고서 | 조달청 제출 | 최종 서명 완료 시 |
| 납품내역서 | 차수별 청구 내역 정리 | 정산 자료 | 필요 시 다운로드 |

### 1.5.5 상태별 가능 작업

| 단계 | 상태 | 가능한 작업 | 불가능한 작업 |
|------|------|------------|--------------|
| 출하 | 대기 | 수정, 삭제, 운송 배차 | - |
| 출하 | 진행중 | 추가변경(수량조정) | 삭제 |
| 출하 | 완료 | 조회만 | 수정, 삭제, 추가변경 |
| 기성차수 | 작성중 | 출하 선택 변경 | - |
| 기성차수 | 서명대기 | 서명 URL 재발송 | 출하 변경 |
| 기성차수 | 확정 | PDF 다운로드, 수금 확인 | 모든 수정 |
| 선급금 | 신청 | 취소 | 재신청 |
| 선급금 | 승인/수금완료 | 조회만 | 모든 수정 |

> **중요**: 기성금 청구에 포함된 출하는 수량 변경이 불가능합니다. 수량 조정이 필요하면 새 출하를 등록해야 합니다.

# 2. 발주 관리

> **용어 안내**: 메뉴에는 **발주관리**로 표기되지만, 실제 화면 상단 제목과 데이터는 조달청 **납품요구** 기준입니다. 본 매뉴얼에서는 "발주(납품요구)"를 같은 의미로 사용합니다.

## 2.1 발주(납품요구) 목록 조회

### 접근 경로
- **메뉴**: 발주관리 → 발주목록
- **화면 주소**: `/admin/order/list`

> 📷 [스크린샷: 발주 목록 화면 — 상단 통계 카드 4개 + 검색줄 + 트리 목록 테이블]

### 화면 구성

**① 상단 통계 카드 (4개)** — 현재 검색 조건 기준으로 자동 집계됩니다.

| 카드 | 의미 |
|------|------|
| 총 납품요구 | 조회된 발주 건수 |
| 금월 납품요구 | 이번 달 납품요구일자 기준 건수 |
| 총 납품요구금액 | 조회 결과의 총 계약금액 합계 |
| 총 예상 원가금액 | 품목 원가 기준 예상 원가 합계 |

**② 검색줄** · **③ 목록 테이블(트리 구조)** 로 구성됩니다.

발주 목록은 **트리 구조**로 표시됩니다.
- **기준(본)계약**: 최상위 행. 왼쪽 화살표(▶/▼) 버튼으로 하위를 펼치고 접습니다.
- **변경계약 / 추가계약**: 본계약 하위에 들여쓰기되어 표시되고, 본계약 행 우측에 **하위 계약 개수 배지**가 붙습니다.

목록 컬럼(좌→우): **No · 납품요구번호 · 납품요구일자 · 수요기관 · 담당자 · 사업명 · 상태 · 건설사 · 총계약금액 · 등록일자**

### 검색 조건

| 검색 항목 | 입력 방식 | 설명 |
|-----------|-----------|------|
| 납품요구일자 | 시작일 ~ 종료일 (달력) | 납품요구일자 기준 기간 |
| 수요기관 | 텍스트 (`수요기관명`) | 발주처 기관명 부분일치 |
| 검색어 | 텍스트 (`프로젝트명, 담당자명`) | 사업명 또는 담당자명으로 검색 |
| 상태 | 드롭다운 | 전체 / 진행중 / 완료 / 취소 |

> 입력 후 **Enter** 키 또는 우측 상단 **[검색]** 버튼으로 조회합니다.

### 단계별 조작 — 조회하기

1. 발주관리 → 발주목록으로 이동합니다.
2. 검색줄에서 기간·수요기관·검색어·상태 중 필요한 조건을 입력합니다. (조건 없이 조회하면 전체가 표시됩니다.)
3. 상단 우측 **[검색]** 버튼을 누릅니다. (검색 중에는 버튼에 회전 아이콘이 표시됩니다.)
4. 결과 테이블 우측 상단의 **[10개씩 / 20개씩 / 50개씩]** 으로 페이지당 표시 개수를 바꿀 수 있습니다.
5. 본계약 행 왼쪽 **▶** 버튼을 눌러 변경/추가계약을 펼쳐 봅니다.
6. 특정 행을 클릭하면 해당 발주의 상세/수정 화면으로 이동합니다.

### 주요 버튼/동작

| 버튼 | 동작 | 조건 |
|------|------|------|
| 검색 | 검색 조건으로 목록 조회 | 항상 |
| 엑셀 | 조회 결과를 Excel 파일로 내려받기 | 항상 (다운로드 중 비활성) |
| 등록 | 발주 등록 화면으로 이동 | **쓰기 권한 있을 때만 표시.** 권한이 없으면 버튼에 "권한이 없습니다" 안내 |

### 오류 메시지 & 대응

| 메시지 | 원인 | 대응 |
|--------|------|------|
| 엑셀 다운로드에 실패했습니다. | 서버 응답 오류·네트워크 문제 | 잠시 후 재시도. 반복되면 관리자 문의 |

## 2.2 발주(납품요구) 등록

### 접근 경로
- 발주목록 → 상단 우측 **[등록]** 버튼
- **화면 주소**: `/admin/order/register`

> 📷 [스크린샷: 발주 등록 화면 — 상단 PDF 업로드 영역 + 기본정보 입력 폼 + 품목 표]

### 등록 방법

발주 등록은 **PDF 자동 추출(권장)** 과 **수동 입력** 두 가지가 있습니다.

#### 방법 1 — PDF 자동 추출 (권장)

1. 화면 상단 **PDF 업로드 영역**에 조달청 납품요구서 **PDF 파일**을 끌어다 놓거나 클릭하여 선택합니다.
   - PDF 외 파일은 받지 않습니다. → 안내: *"PDF 파일만 업로드 가능합니다."*
2. 업로드되면 자동으로 데이터 추출이 진행됩니다. → 진행 표시: *"PDF 파일 업로드 및 데이터 추출 중..."*
3. 추출이 끝나면 안내가 표시됩니다. → *"PDF 데이터 추출 완료. 계약 유형을 선택해주세요."*
4. **계약 유형**을 선택합니다.
   - **본계약**(최초) / **변경계약**(기존 계약 내용 변경) / **추가계약**(별도 차수 추가)
5. 추출된 계약 기본정보와 품목 목록을 확인하고 필요 시 수정합니다.
6. **모든 품목의 납품기한**을 입력합니다. (누락 시 저장 불가 → *"모든 품목의 납품기한을 입력해주세요."*)
7. **[저장]** 을 누릅니다. → 성공: *"계약이 성공적으로 등록되었습니다."*

> 같은 납품요구번호(-00)를 다시 업로드하면 신규 등록 대신 **기존 건 업데이트** 경로로 처리됩니다. → *"기존 납품요구가 업데이트되었습니다."* (기존 ID를 찾지 못하면 *"기존 납품요구 ID를 확인할 수 없습니다. 목록에서 직접 수정해 주세요."*)

#### 방법 2 — 수동 입력

1. 기본 정보를 직접 입력합니다(아래 입력 필드 표 참고).
2. 품목 영역에서 **[품목 추가]** 로 행을 추가하고 품목명·규격·수량·단가를 입력합니다.
3. 모든 품목의 납품기한을 입력합니다.
4. **[저장]** 을 누릅니다.

### 입력 필드 설명

| 필드 | 필수 | 설명 |
|------|:----:|------|
| 계약번호 | O | 계약 고유 번호 |
| 계약일자 | O | 계약 체결일 |
| 납품요구번호 | O | 조달청 발행 번호(예: `R26TB0...-00`). 끝 `-00`은 본계약, `-01` 이상은 변경계약 |
| 납품요구일자 | O | 조달청 납품요구일 |
| 사업명 | O | 현장/사업명 |
| 수요기관명 | O | 발주처 기관명 |
| 계약 유형 | O | 본계약 / 변경계약 / 추가계약 |
| 품목별 납품기한 | O | 품목마다 입력(전체 입력해야 저장 가능) |

### 주의사항

> **변경계약·추가계약 등록 시** 반드시 어느 본계약에 연결되는지 확인하세요. 같은 납품요구번호 체계(-00 → -01)로 연결되어 목록 트리에 하위로 표시됩니다. 연결이 끊기면 별도 본계약으로 잘못 등록될 수 있습니다.

## 2.3 발주(납품요구) 수정

### 접근 경로
- 발주목록에서 해당 행 클릭 → 상세/수정 화면
- **화면 주소**: `/admin/order/edit/{발주ID}` (상세는 `/admin/order/detail/{발주ID}`)

> 📷 [스크린샷: 발주 수정 화면 — 기본정보·품목 표·원본 PDF 보기 버튼]

### 단계별 조작 — 수정하기

1. 발주목록에서 수정할 발주 행을 클릭합니다.
2. 내용을 수정합니다. (출하가 시작된 품목은 수량 축소가 제한됩니다 — 아래 표 참고.)
3. **[수정]** 또는 **[저장]** 을 누릅니다. → 성공: *"납품요구가 수정되었습니다."*
4. 원본 납품요구서를 확인하려면 **[원본 PDF 보기]** 를 누릅니다.

### 수정 가능 / 불가 규칙

| 항목 | 규칙 |
|------|------|
| 기본 정보 | 출하 전까지 자유 수정 |
| 품목 수량 | **기출하 수량 이상**으로만 수정 가능(이미 나간 물량보다 작게 줄일 수 없음) |
| 품목 삭제 | 이미 출하된 품목은 삭제 불가 |
| 금액 | 기성금 청구가 끝난 건은 청구에 영향 주는 금액 변경 불가 |
| 납품완료 처리분 | 핵심 정보 변경 불가 |

### 오류 메시지 & 대응

| 메시지 | 원인 | 대응 |
|--------|------|------|
| 납품요구 수정에 실패했습니다. | 검증 실패·서버 오류 | 입력값(특히 수량/기한) 확인 후 재시도 |
| 원본 PDF 를 여는 중 오류가 발생했습니다. | 원본 PDF 파일 경로 문제 | 관리자 문의(원본 보관 상태 확인 필요) |
| 연결된 자금 정보가 없습니다. | 발주에 자금 레코드 미생성 | 자금관리 화면에서 자금 정보 확인 |
| 수금 확인 처리 중 오류가 발생했습니다. | 수금 확인 처리 실패 | 잠시 후 재시도, 반복 시 관리자 문의 |

# 3. 출하 관리

## 3.1 출하 목록 조회

### 접근 경로
- **메뉴**: 출하관리 → 출하목록
- **화면 주소**: `/admin/shipping/list`

> 📷 [스크린샷: 출하 목록 화면 — 검색줄 + 출하 목록 테이블(합지/추가 배지 포함)]

### 화면 구성

출하 목록은 각 출하 건의 상태·수량·금액을 한 줄로 보여줍니다. 일부 행에는 다음 배지가 표시됩니다.
- **합지**: 여러 발주 품목이 한 출하로 합쳐진 건
- **추가**: 추가변경으로 품목/수량이 더해진 건

목록 컬럼(좌→우): **출하NO · 납품요구일자 · 수요기관 · 사업명 · 출하일자 · 상태 · 출고 · 출하수량 · 출하금액 · 비고**

### 출하 상태

| 상태 | 설명 | 표시 색상 |
|------|------|-----------|
| 등록 | 출하 정보 등록 완료 | 회색 |
| 운송중 | 배송 출발 | 파란색 |
| 배송완료 | 현장 도착 | 주황색 |
| 납품확인 | 납품확인 완료 | 녹색 |

### 검색 조건

| 검색 항목 | 입력 방식 | 설명 |
|-----------|-----------|------|
| 출하일자 | 시작일 ~ 종료일 (달력) | 출하일자 기준 기간 |
| 납품요구번호 | 텍스트 (`납품요구번호 입력 또는 조회`) | 연결된 발주의 납품요구번호 |
| 검색어 | 텍스트 (`수요기관, 사업명, 출하NO`) | 수요기관·사업명·출하번호 통합 검색 |
| 상태 | 드롭다운 | 등록 / 운송중 / 배송완료 / 납품확인 |

### 단계별 조작 — 조회하기
1. 출하관리 → 출하목록으로 이동합니다.
2. 출하일자 기간·납품요구번호·검색어·상태 중 필요한 조건을 입력합니다.
3. **[검색]** 또는 입력 후 **Enter** 로 조회합니다.
4. 행을 클릭하면 해당 출하의 상세/수정 화면으로 이동합니다.

### 주요 버튼/동작

| 버튼 | 동작 | 조건 |
|------|------|------|
| 검색 | 검색 조건으로 목록 조회 | 항상 |
| 엑셀 | 조회 결과를 Excel로 내려받기 | 항상 |
| 등록 | 출하 등록 화면으로 이동 | 쓰기 권한 있을 때만(없으면 "권한이 없습니다") |

### 오류 메시지 & 대응

| 메시지 | 원인 | 대응 |
|--------|------|------|
| 등록된 출하 정보가 없습니다. | 조건에 맞는 출하 없음 | 검색 조건 완화 |
| 엑셀 다운로드에 실패했습니다. | 서버/네트워크 오류 | 잠시 후 재시도 |

## 3.2 출하 등록

### 접근 경로
- 출하목록 → 상단 우측 **[등록]** 버튼
- **화면 주소**: `/admin/shipping/register`

> 📷 [스크린샷: 출하 등록 화면 — 발주 선택 + 자동 채움 기본정보 + 품목/수량 입력 표]

### 등록 절차

#### Step 1 — 발주(납품요구) 선택
1. **납품요구번호(발주번호)** 필드의 **[조회]** 로 발주를 검색·선택합니다. (`납품요구번호를 선택하세요`)
2. 발주를 선택하면 **납품요구일자·사업명·수요기관명·담당자·건설사** 등이 **자동으로 채워집니다.** (`발주 선택 시 자동으로 입력됩니다`)
   - 발주 정보를 불러오지 못하면 → *"발주 정보를 불러오는데 실패했습니다."*

#### Step 2 — 출하 품목 및 수량 입력
1. 발주 품목 중 출하할 품목을 추가합니다.
   - 이미 추가한 품목을 또 추가하면 → *"이미 추가된 품목입니다."*
2. 각 품목의 **출하 수량**을 입력합니다.
   - 0 이하 등 잘못된 값 → *"올바른 수량을 입력하세요."*
   - 잔여수량 초과 → *"출하수량은 잔여수량을 초과할 수 없습니다."* (출하수량 ≤ 발주수량 − 기출하수량)

#### Step 3 — 공급원·운송·기타 정보
1. **공급원**(제조사/OEM)을 선택합니다. **(필수)**
2. **출하일자**를 선택합니다. **(필수)**
3. 배송비, 비고 등 부가 정보를 입력합니다.

#### Step 4 — 저장
1. 입력 내용을 확인합니다.
2. **[저장]** 을 누릅니다.
   - 출하 품목이 없으면 → *"출하할 품목이 없습니다. 출하수량을 입력해주세요."*
   - 성공 → *"출하 정보가 저장되었습니다."*

### 입력 필드 설명

| 필드 | 필수 | 설명 |
|------|:----:|------|
| 납품요구번호(발주번호) | O | 출하할 발주. [조회]로 선택 |

# 4. 납품 확인

## 4.1 납품확인 개요

납품확인은 현장에서 실제 물품을 수령하고 확인하는 과정입니다. 현장 담당자(인수자)가 **모바일 기기**로 직접 처리합니다.

### 납품확인 프로세스

```
출하 (배송완료 상태)
       ↓  관리자: 모바일 URL 발송 (SMS/LMS)
현장 담당자 URL 접속 (로그인 불필요, 토큰 인증)
       ↓
정보·수량 확인 → (사진 촬영) → 터치 서명 저장
       ↓
납품확인 완료
```

> **두 가지 진행 모드가 있습니다.**
> - **일반(full) 모드**: 현장 담당자가 **사진 업로드 + 서명**을 모두 진행
> - **서명전용(signature) 모드**(URL `?mode=signature`): 사진은 **관리자가 미리 업로드**하고, 현장 담당자는 **서명만** 진행
>
> 화면에 *"납품 사진은 관리자가 처리했습니다. 서명만 진행해주세요."* 가 보이면 서명전용 모드입니다.

## 4.2 관리자 화면 (PC)

### 접근 경로
- **메뉴**: 납품관리 → 납품확인
- **화면 주소**: `/admin/delivery/list`

> 📷 [스크린샷: 납품확인 목록(PC) — 발주 단위 트리 + 출하별 납품확인 상태]

### 화면 구성
납품확인 목록은 **트리 구조**입니다. 발주 단위로 그룹화되고, 각 발주 하위에 출하 건과 납품확인 상태가 표시됩니다.

### 단계별 조작 — 납품확인 URL 발송
1. 납품확인이 필요한 출하 건을 선택합니다.
2. **[납품확인 URL 발송]** 을 누릅니다.
3. **수신자명**과 **휴대폰 번호**를 입력합니다.
4. 메시지 유형(**SMS / LMS**)을 선택합니다.
5. **[발송]** 을 누릅니다.

### URL 유효기간
- 발송된 URL은 **24시간** 동안 유효합니다.
- 만료된 링크로 접속하면 *"서명 링크가 만료되었습니다. 관리자에게 새 서명 링크를 요청해주세요."* 가 표시됩니다. → 관리자가 **재발송**합니다.

## 4.3 모바일 납품확인 (현장 담당자)

### 접속 방법
- 문자로 받은 URL을 터치하면 모바일 브라우저에서 납품확인 페이지가 열립니다.
- **별도 로그인 불필요**(토큰 기반 인증).
- **화면 주소**: `/m/delivery/{토큰}` (서명전용은 `?mode=signature`)

> 📷 [스크린샷: 모바일 납품확인 화면 — 배송정보 + 품목표 + 사진/서명 영역]

### 납품확인 절차

#### Step 1 — 정보 확인
화면 상단에서 **운송장번호 · 차량번호 · 배송지 · 배송예정일 · 현장소장**과 품목 표(품목·규격·수량·비고)를 확인합니다.

#### Step 2 — 사진 (일반 모드)
1. **추가 사진 업로드** 영역에서 사진을 촬영/선택합니다. **(최대 5장)**
2. 첨부된 사진은 탭하여 크게 볼 수 있고, 잘못 올린 사진은 삭제할 수 있습니다.
   - 삭제 시 확인: *"이 사진을 삭제하시겠습니까?"* / 실패 시 *"삭제 실패: …"*
- 서명전용 모드에서는 이 영역 대신 *"첨부된 사진이 없습니다. 관리자가 이후 업로드할 예정입니다."* 가 표시될 수 있습니다.

#### Step 3 — 서명
1. **현장소장 서명** 영역에 손가락으로 서명합니다.
2. **[서명 저장]** 을 누르면 자동으로 완료 처리됩니다. (*"서명 후 서명 저장 버튼을 누르시면 자동으로 완료됩니다."*)
3. 서명 저장 시 **GPS 위치**가 함께 기록됩니다. (위치 권한이 없거나 실패해도 서명은 정상 저장됩니다 — 위치는 선택사항)
4. 성공하면 *"서명이 저장되었습니다"* 화면이 표시됩니다.

### 오류 메시지 & 대응

| 메시지 | 원인 | 대응 |
|--------|------|------|
| 서명 링크가 만료되었습니다. | URL 24시간 경과 | 관리자에게 재발송 요청 |
| 서명 저장에 실패했습니다. | 네트워크/서버 오류 | 통신 확인 후 재시도 |
| 삭제 실패: … | 사진 삭제 처리 오류 | 잠시 후 재시도 |

# 5. 납품완료계

## 5.1 납품완료계 개요
납품완료계는 모든 납품이 완료된 후 **조달청에 제출하는 공식 문서**입니다. 현장소장·감리원 서명을 받아 완성합니다.

### 납품완료계 상태 흐름
```
PENDING (대기)
   ↓ 첫 출하 생성
IN_PROGRESS (납품중)
   ↓ 모든 납품확인 완료
PENDING_SIGNATURE (서명 대기)
   ↓ 현장소장 서명
PARTIAL_SIGNED (일부 서명)
   ↓ 감리원 서명
COMPLETED (완료) → 3종 PDF 자동 생성
   ↓ 조달청 제출
SUBMITTED (제출완료)
```

## 5.2 납품완료계 목록
### 접근 경로
- **메뉴**: 납품관리 → 납품완료계
- **화면 주소**: `/admin/delivery-done/list`

목록 컬럼: 납품요구번호 · 현장명 · 상태 · 납품확인 현황(예: 5/5) · 서명 현황(현장소장/감리원) · 등록일

## 5.3 납품완료계 상세 화면
### 접근 경로
- 목록에서 행 클릭 → **화면 주소**: `/admin/delivery-done/detail/{ID}`

> 📷 [스크린샷: 납품완료계 상세 — 금액 요약 + 품목 표 + 사진 관리 + 변경 이력]

### 화면 구성
- **금액 요약**: **품목총액**(품대계 = 매출 기준 금액) + **수수료**(참고 표기)
- **품목 표** 컬럼: 순번 · 품명 · 규격 · 단위 · 단가 · 발주수량 · 추가수량 · 유효수량 · 납품수량 · 잔여수량 · 완료. 추가분 품목에는 **추가** 배지가 붙습니다.
- **변경 이력**: 일시 · 유형 · 품목 · 증감 기록

### 사진 관리 (사진대지용)
납품완료계 상세에서 현장 사진을 관리하고, **사진대지 PDF에 들어갈 사진**을 지정합니다.

| 동작 | 설명 | 안내/제약 |
|------|------|-----------|
| 사진 교체 | 기존 사진을 다른 사진으로 교체 | 성공: "사진이 교체되었습니다." |
| 사진 추가 | 사진을 추가 업로드 | "사진이 추가되었습니다. 사진대지에 포함하려면 '사진 선택'에서 지정하세요." |
| 사진대지 선택 | 사진대지에 넣을 사진 지정 | **출하당 최대 2장**, **최소 1장**. 저장 성공: "사진대지 선택이 저장되었습니다." |

- 최대 초과: *"사진대지는 출하당 최대 2장까지입니다. 다른 사진의 '대지'를 먼저 해제한 후 선택해 주세요."*
- 최소 미달: *"출하당 최소 1장을 선택해야 합니다."*
- 사진을 교체/추가/재선택하면 **사진대지 PDF가 자동으로 다시 생성**됩니다.

### 재계산 (관리자 전용)
출하 이관 등으로 수량/금액이 어긋났을 때, 상세 화면의 **[재계산]** 버튼(SYSTEM_ADMIN 권한)으로 납품완료 수량·금액을 다시 맞춥니다. 확인 창 후 실행되며, 성공 시 *"재계산이 완료되었습니다."*, 실패 시 *"재계산에 실패했습니다."* 가 표시됩니다.

## 5.4 서명 수집
### 단계별 조작
1. 납품완료계 상세에서 **[서명 URL 발송]** 을 누릅니다.
2. 서명자를 선택합니다. **현장소장 → 감리원 순서**입니다.
3. 메시지 미리보기(납품요구번호·수요기관·사업명·서명 URL)를 확인하고 **[발송]** 합니다.
4. 서명 현황은 **대기(⏳) / 완료(✅)** 로 표시됩니다. URL 유효기간은 **24시간**입니다.

## 5.5 PDF 문서 생성 및 다운로드
**현장소장 + 감리원** 양측 서명이 완료되면 3종 PDF가 자동 생성됩니다.

| 문서 | 설명 |
|------|------|
| 납품확인서 | 품목별 납품 내역 확인 문서 |
| 납품완료계 | 최종 납품완료 보고서(조달청 제출용) |
| 사진대지 | 사진대지로 지정한 현장 사진 모음(출하당 1~2장) |

다운로드: 상세 화면의 **문서** 영역에서 문서별 **[다운로드]**, 또는 **[전체 다운로드]**(ZIP).

## 5.6 조달청 제출
1. 모든 서명 완료와 PDF 생성을 확인합니다.
2. **[제출 완료 처리]** 를 누르면 상태가 **SUBMITTED(제출완료)** 로 바뀝니다.

> **주의**: 제출 완료 처리 후에는 수정이 불가능합니다.

# 6. 자금 관리 - 선급금

> 이 장은 **선급금**(계약 체결 후 납품 전에 선지급받는 금액)의 신청·문서 다운로드·수금 확인 절차를 다룹니다.
> 독자: 회계·자금 담당 실무자. 화면 조작 중심으로 설명합니다.

## 6.1 선급금 개요

선급금은 계약 체결 후 납품 전에 **선지급**되는 금액입니다. 통상 계약금액(품대계)의 **70%**를 선급금으로 지급받습니다.

- **기본 비율**: 70% (신청 모달에서 0~100% 범위로 수정 가능)
- **신청 횟수**: 발주(자금) 1건당 **1회만** 신청 (수금확인 전이면 삭제 후 재신청 가능)
- **금액 기준**: 계약총액 = `orders.item_total_amount`(품대계). 매출·집계와 동일 기준.
- **선급금은 서명 절차가 없습니다.** 신청 즉시 5종 PDF가 자동 생성됩니다.

### 선급금 계산식

```
선급금 예상액 = 계약총액(품대계) × 선급금 비율(기본 70%)
```

예: 계약총액 1억 원 → 70% → 선급금 예상액 **7,000만 원**

> 코드 근거: `AdvancePaymentModal.vue` `expectedAdvanceAmount = Math.round(totalContractAmount × rate/100)`,
> `[id].vue` `expectedAdvanceAmount = Math.floor(contractTotalAmount × rate /100)` (탭 표시는 내림 처리).

### 선급금 프로세스

```
선급금 신청 (비율·금액 입력)
    ↓
5종 PDF 문서 자동 생성
    ↓
상태: REQUESTED (신청)
    ↓
실제 입금 → 입금(수금) 확인 처리
    ↓
상태: PAID (수금완료)
```

> ※ 화면상 선급금에는 별도의 "승인" 버튼이 없습니다. 신청(REQUESTED) 또는 승인(APPROVED) 상태에서 **[입금 확인하기]** 버튼으로 바로 수금 처리합니다. (APPROVED 상태는 백엔드 데이터로는 존재하나, 화면에서 승인 버튼을 통한 별도 단계는 노출되지 않습니다.)

### 선급금 상태값

| 상태 코드 | 화면 라벨 | 의미 |
|-----------|-----------|------|

## 6.3 선급금 신청

### 접근 위치

자금 상세 화면 → **선급금** 탭 → 우측 상단 **[선급금 신청하기]** 버튼

### 단계별 조작

1. 자금 상세 화면에서 **선급금** 탭 선택 (기본 선택됨)
2. 우측 상단 **[선급금 신청하기]** 버튼 클릭
   - 이미 신청한 경우 버튼은 **[선급금 신청완료]** 로 바뀌고 비활성화됩니다.
   - 신청 조건을 충족하지 못하면 버튼이 비활성화되며, 마우스를 올리면 사유가 툴팁으로 표시됩니다.
3. **선급금 신청** 모달 창이 열립니다.
4. 상단 요약에서 **총 계약금액**(수정 불가)·**선급금 비율**(수정 가능)을 확인하면, **선급금 예상액**이 자동 계산됩니다.
5. **선급금 비율**을 조정하면 **신청 금액**이 예상액으로 자동 채워집니다. (필요 시 신청 금액 직접 수정 가능)
6. **신청일**을 선택합니다. (기본값: 오늘)
7. (선택) **비고**를 입력합니다.
8. 하단 **[선급금 신청]** 버튼 클릭
9. "선급금 신청이 완료되었습니다." 결과 모달이 표시되고, 서버가 **5종 PDF**를 자동 생성합니다.
10. 상태: **REQUESTED (신청)**

> 📷 [스크린샷: 선급금 신청 모달 — 총 계약금액/선급금 비율(입력)/선급금 예상액 + 신청 금액·신청일·비고 폼]

### 입력 필드

| 필드 | 의미 | 필수 | 제약 | 비고 |

# 7. 자금 관리 - 기성금

> 이 장은 **기성금**(납품 진행에 따라 단계별로 청구하는 금액)의 청구·문서 발행·수금 확인 절차를 다룹니다.

## 7.1 기성금 개요

기성금은 납품이 진행됨에 따라 **차수별(1차, 2차, 3차…)로 청구**하는 금액입니다. **납품확인(인수증 서명)이 완료된 출하**를 선택하여 기성금을 청구합니다.

- **청구 단위**: 차수(Baseline). 발주 1건에 여러 차수.
- **청구 대상**: 납품확인 완료된 출하 중 아직 기성 청구에 포함되지 않은 출하.
- **금액 산정**: 선택한 출하의 금액을 합산하되, **전량 출하된 품목은 원수량(order_items)으로 정합(cap)** 하여 저장값과 일치시킵니다(미리보기에 자동 반영). 따라서 "출하금액 단순합"과 "실제 청구금액"이 다를 수 있습니다.
- **다량납품할인**: 단가합(룰 기준) ≥ 1억 2%, ≥ 2억 3%(조달청 원단위 절사). 표준 발주에만 자동 적용.
- **OEM 지급 예정액**: 청구금액 × 선급금 비율(기본 70%).

### 기성금 계산식

```
선택 출하 합계 = Σ(선택한 출하의 totalAmount)
청구금액(저장값) = 원수량 정합(cap) 적용 합계   ← 미리보기 = 백엔드 저장값
다량납품할인액 = 청구금액 − (청구금액 × (1−할인율)) 의 원단위 절사
할인 후 청구액 = 청구금액 − 다량납품할인액
선급금 차감액 = MIN(할인 후 청구액 × 선급금율, 미정산 선급금 잔액)
실수금액 = 할인 후 청구액 − 선급금 차감액
OEM 지급 예정액 = 청구금액 × 선급금율(기본 70%)
```

> 금액 정책: 매출·집계 기준은 품대계(`item_total_amount`)이며, 기성 차수 청구는 별도 회계 규칙(원수량 정합 cap + 다량납품할인)이 적용됩니다.
> 코드 근거: `ProgressPaymentModal.vue` (claimAmount/reconcileDiff/bulkDiscount*/advanceDeductionAmount/actualReceivableAmount/oemPaymentAmount), `baseline.service.ts previewBaseline`.

### 기성금 프로세스 (현재 구현: 서명 없이 즉시 발행)

```
납품확인(인수증 서명) 완료된 출하 선택
         ↓
기성 청구 (차수 생성)
         ↓
서명란 공란 납품확인서 + 사진대지 PDF 즉시 발행
         ↓
(현장 서명을 종이로 받은 경우) 서명본 스캔 PDF 업로드 → 다운로드 시 스캔본 우선
         ↓
실제 입금 후 수금 확인
         ↓
상태: PAID
```

> ⚠️ **중요(코드 vs 지식베이스 불일치)**: 현재 화면 구현은 **서명 절차 없이** 기성 청구 즉시 서명란 공란 납품확인서 PDF를 발행하고, 종이로 받은 서명본을 스캔 업로드하는 방식입니다(`ProgressPaymentModal.submitClaim → baselineService.createBaselineDirect`, 안내: "기성 청구가 완료되었습니다. 납품확인서 PDF를 다운로드할 수 있습니다."). 지식베이스(`domain-rules.yaml progress_payment.requires_signature: true`)와 과거 매뉴얼의 "서명 URL 발송 → 현장소장→감리원 서명" 흐름은 **모바일 서명 발송 방식(레거시/대체 경로)** 으로, 코드상 `ProgressSignatureModal`·`createAndSendSignature`·`sendSignatureUrl` 형태로 남아 있습니다. 7.3에 두 흐름을 모두 정리합니다. 운영에서 실제 사용 중인 경로는 `(확인 필요)`.

## 7.2 기성금 청구

### 접근 위치

자금 상세 화면 → **기성금** 탭 → 우측 상단 **[기성 청구하기]** 버튼

### 청구 조건

- **납품확인(인수증 서명)이 완료된 출하**만 청구 가능
- 이미 기성 청구에 포함된 출하는 목록에 나타나지 않음(선택 불가)
- 납품완료(COMPLETED)된 발주는 기성 청구 불가
- 마지막 1건은 납품완료계용으로 남겨야 할 수 있음(아래 선택 제한 참조)

> 코드 근거: `[id].vue canRequestProgress / getProgressButtonTooltip`, `baseline.service.ts getAvailableShipments`.

### 단계별 조작

#### Step 1. 기성 청구 시작

1. 자금 상세 화면 → **기성금** 탭
2. 우측 상단 **[기성 청구하기]** 버튼 클릭
   - 비활성 시 툴팁으로 사유 표시(아래 표 참조)

#### Step 2. 청구 가능 출하 선택

> 📷 [스크린샷: 기성 청구 모달 — 청구 가능 출하 목록(체크박스) + 청구 금액 계산 섹션 + 품목별 정합 미리보기]

1. **청구 가능 출하 목록**이 표시됩니다(기본적으로 선택 가능 출하가 자동 선택됨).
2. 청구할 출하를 체크박스로 선택/해제합니다(행 클릭으로도 토글). 헤더 체크박스로 전체 선택/해제.
3. 선택 제한이 있는 경우 상단에 경고가 표시됩니다:
   > "마지막 출하는 납품완료계로 처리해야 합니다. 최대 N건까지 선택 가능합니다."
4. 선택에 따라 우측/하단 계산이 실시간 갱신됩니다:
   - **선택한 출하 수 / 선택 출하 합계**
   - **(-) 원수량 정합 (전량 출하분)** 또는 (+) 정합
   - **청구 금액** (원수량 정합 적용된 저장 예정값)
   - **품목별 금회 청구(원수량 정합)** 표: 품목 / 금회수량 / 잔여수량 / 금액
   - **다량납품할인(자동)** 미리보기: 단가합·적용율·할인 전 청구·(-)할인·(=)할인 후 청구액
   - **선급금 차감 계산**(선급금 신청한 경우): 비율·미정산 잔액·(-)차감액·(=)실수금액
   - **OEM 지급 예정 금액**
5. (선택) **비고** 입력

#### Step 3. 청구 확정 (서명 없이 즉시 발행)

1. 하단 **[기성 청구]** 버튼 클릭
2. 차수가 생성되고 **서명란 공란 납품확인서 + 사진대지 PDF**가 즉시 발행됩니다.
3. 알림: "기성 청구가 완료되었습니다. 납품확인서 PDF를 다운로드할 수 있습니다."
4. 기성금 탭 목록에 새 차수가 추가됩니다.

> 코드 근거: `ProgressPaymentModal.vue submitClaim`, `baseline.service.ts createBaselineDirect (createWithoutSignature)`.

### 입력 필드 (기성 청구 모달)

| 필드 | 의미 | 필수 | 제약 | 비고 |
|------|------|:---:|------|------|
| 출하 선택(체크박스) | 청구할 출하 | 필수 | 최소 1건, 선택 제한 시 최대 N건 | 미선택 시 청구 불가 |
| 비고 | 청구 관련 메모 | 선택 | - | - |
| (자동) 청구 금액 | 원수량 정합 청구액 | - | 자동 | 미리보기=저장값 |
| (자동) 다량납품할인 | 단가합 기준 0/2/3% | - | 표준 발주만 | 원단위 절사 |
| (자동) 선급금 차감액 | MIN(할인후청구×율, 미정산잔액) | - | 선급금 신청 시 | - |

### 버튼/동작 (기성 청구 모달)

| 버튼 | 동작 | 조건 |
|------|------|------|
| [기성 청구하기] | 청구 모달 열기 | 청구 가능 조건 충족 시 활성 |
| 헤더 체크박스 | 전체 선택/해제 | 선택 제한 시 최대 수까지만 |
| [기성 청구] | 차수 생성 + 공란 납품확인서·사진대지 PDF 발행 | 출하 1건 이상 선택 |
| [취소] | 모달 닫기 | 항상 |

### 사용 예시

> 발주 계약총액 1억 원, 선급금 70%(7,000만 원) 신청·수금 완료. 1차로 납품확인 완료된 출하 2건(합계 4,000만 원) 선택.
> - 전량 출하 품목 없음 → 청구금액 4,000만 원
> - 다량납품할인: 단가합 1억 미만이면 미적용(0%) / 1억 이상이면 2% 등 자동
> - 선급금 차감액 = MIN(4,000만×70%=2,800만, 미정산 잔액 7,000만) = 2,800만 원
> - 실수금액 = 4,000만 − 2,800만 = **1,200만 원**
> - OEM 지급 예정액 = 4,000만 × 70% = 2,800만 원

## 7.3 서명 흐름

기성 차수의 납품확인서 서명에는 두 가지 경로가 코드에 존재합니다.

### (A) 현재 화면 기본 경로 — 서명 없이 즉시 발행 + 스캔 업로드

- 기성 청구 시 서명란이 **공란**인 납품확인서 PDF를 즉시 발행.
- 현장에서 종이로 서명을 받은 뒤, 기성금 탭의 **[스캔업로드]** 버튼으로 서명본 스캔 PDF(.pdf, 최대 20MB)를 업로드.
- 이후 다운로드 시 **스캔본이 우선** 제공됩니다.

> 코드 근거: `FundProgressTab.vue` (납품확인서/사진대지/스캔업로드 버튼), `baseline.service.ts uploadConfirmationScan`.

### (B) 레거시/대체 경로 — 모바일 서명 URL 발송

> ⚠️ 아래는 코드에 존재하는 모바일 서명 방식입니다. 현재 운영 사용 여부는 `(확인 필요)`.

- 기성 청구 시 현장소장·감리원에게 서명 URL이 포함된 메시지(LMS/SMS) 발송.
- **서명 순서**: ① 현장소장 → ② 감리원.
- 양측 서명 완료 시 납품확인서·사진대지 PDF 자동 생성.

#### 서명 상태값

| 상태 코드 | 화면 라벨 | 의미 |
|-----------|-----------|------|
| PENDING_SIGNATURE | 서명대기 | 발송 후 서명 대기 |
| PARTIAL_SIGNED | 일부서명 | 현장소장 또는 감리원 중 1명 서명 |
| SIGNATURE_COMPLETED | 서명완료 | 양측 모두 서명 완료 |

서명 URL 유효시간: 24시간(만료 시 재발송 필요). 만료/중복 서명 시 모바일에서 "서명 링크가 만료되었습니다." / "이미 서명이 완료되었습니다." 등의 메시지가 표시됩니다.

> 코드 근거: `types/baseline.ts SIGNATURE_STATUS_LABELS`, `baseline.service.ts sendSignatureUrl / baselineMobileService.uploadSignature`, `domain-rules.yaml signature_rules.baseline_signature (url_validity_hours: 24)`.

## 7.4 기성금 문서 다운로드 / 스캔 업로드

기성금 탭의 각 차수 행 **PDF** 열에서:

- **[납품확인서]**: 납품확인서 PDF(서명란 공란) 미리보기/다운로드
- **[사진대지]**: 사진대지 PDF
- **[스캔업로드]**: 서명받은 종이 납품확인서 스캔본(PDF) 업로드

스캔 업로드 시:
- PDF 파일만 허용(MIME 또는 .pdf 확장자). 아니면 "PDF 파일만 업로드할 수 있습니다."
- 최대 20MB. 초과 시 "파일 크기는 20MB를 초과할 수 없습니다."
- 성공 시 "서명본 스캔이 업로드되었습니다. 이후 다운로드 시 스캔본이 제공됩니다."

> 코드 근거: `FundProgressTab.vue onScanFileSelected`, `baseline.service.ts uploadConfirmationScan`.

## 7.5 기성금 수금 확인

### 단계별 조작

1. 기성금 탭 목록에서 해당 차수 행 확인
2. 상태가 **승인(APPROVED)** 인 행의 **수금확인** 열에서 **[수금확인]** 버튼 클릭
   - 상태가 PAID면 "완료"로 표시, 그 외에는 "-" 표시
3. **기성금 입금확인** 모달에서 정보 입력:

| 필드 | 의미 | 필수 | 제약 | 비고 |
|------|------|:---:|------|------|
| 입금일 | 실제 입금 날짜 | 필수 | 오늘 이전(max=today) | 기본값 오늘 |
| 입금 금액 | 실제 입금된 금액 | 필수 | 0보다 커야 함 | [전액 입력] = 실수금액(선급금 차감 시) 또는 승인/신청액 |
| 입금 계좌 | 입금 계좌 정보 | 선택 | - | - |
| 비고 | 추가 메모 | 선택 | - | - |

4. (선급금 차감이 있는 경우) 모달에 **선급금 차감액 / 실수금액(입금 예정액)** 과 **정산 현황(수금 확인 후 선급금 정산 누계·미정산 잔액)** 이 함께 표시됩니다.
5. **[수금 확인]** 버튼 클릭 → 성공 메시지 후 자동 닫힘
6. 상태: **PAID (수금완료)**

> 📷 [스크린샷: 기성금 입금확인 모달 — 기성금 금액/(-)선급금 차감액/(=)실수금액 + 정산 현황 + 입금일·입금 금액·입금 계좌·비고]

> 코드 근거: `FundProgressTab.vue` (수금확인 버튼, status==='APPROVED'), `CollectionConfirmModal.vue` (paymentType='progress', 선급금 차감/정산 현황), `fund.service.ts confirmPayment`.

### 기성금 요청 상태값

| 상태 코드 | 의미 | 화면 동작 |
|-----------|------|-----------|
| REQUESTED | 신청 | 수금확인 열 "-" |
| APPROVED | 승인 | **[수금확인]** 버튼 노출 |
| PAID | 수금완료 | "완료" 표시 |
| REJECTED | 반려 | "-" |

> 코드 근거: `types/fund.ts PaymentStatus`, `FundProgressTab.vue`.

## 7.6 기성금 이력 조회

기성금 탭의 표에서 모든 기성 차수 이력을 확인할 수 있습니다(잔금 BALANCE 타입은 제외하고 표시).

| 열 | 의미 |
|----|------|
| 차수 | 1차, 2차, 3차… (`paymentSeq`) |
| 청구일 | 기성 청구 날짜 |
| 청구금액 | 해당 차수 청구 금액 |
| 선급금차감 | (선급금 신청 시) 차감액(빨강, 음수) |
| 실수금액 | (선급금 신청 시) 청구금액 − 선급금차감 |
| 수금일 | 실제 입금 날짜 |
| 상태 | 신청/승인/수금완료/반려 |
| PDF | 납품확인서/사진대지/스캔업로드 |
| 수금확인 | [수금확인] 버튼 / 완료 / - |

이력이 없으면 "기성금 이력이 없습니다." 가 표시됩니다.

> 코드 근거: `FundProgressTab.vue` 테이블 헤더 및 행 바인딩.

# 8. 자금 관리 - 잔금 및 OEM

자금 관리는 발주(납품요구) 1건당 1개의 자금 정보를 두고 **선급금 → 기성금 → 잔금** 의 순서로 수금을 관리하며, 협력업체(OEM 제조사)에 지급하는 **OEM 지급** 까지 함께 다룹니다. 이 장은 자금 관리의 마지막 단계인 **잔금** 과 **OEM 지급** 을 중심으로 설명합니다. (선급금·기성금 신청·수금은 7장 참조)

> 📷 [스크린샷: 자금 관리 상세 화면 전체 - 상단 요약 카드 + 탭(선급금/기성금/잔금/OEM 지급)]

## 8.0 자금 상세 화면 들어가기

| 구분 | 내용 |
|------|------|
| **메뉴 경로** | 자금 관리 → 목록에서 해당 건 클릭 |
| **목록 라우트** | `/admin/funds` |
| **상세 라우트** | `/admin/funds/{자금ID}` |
| **접근 권한** | 시스템관리자, 리드파워담당자 |

자금 상세 화면은 상단의 **요약 카드 + 수금 진행 바**, 그 아래의 **탭 영역**(선급금 / 기성금 / 잔금 / OEM 지급)으로 구성됩니다.

> 📷 [스크린샷: 자금 상세 상단 요약 카드 - 계약총액 / 선급금(미정산 표시) / 잔금 / 수금 진행 바]

## 8.1 잔금 개요

잔금은 **납품완료 후** 정산되는 나머지 금액입니다. 계약의 마지막 수금 단계로, 발주 1건당 **1회만** 등록할 수 있습니다.

### 잔금 계산식

```
잔금 = 계약총액 − 선급금 − 기성금누계
```

시스템은 잔금을 두 가지 기준으로 동시에 계산해 둡니다. (백엔드 `BalanceInfo` 기준)

| 계산 기준 | 코드값 | 계산식 | 설명 |
|-----------|--------|--------|------|
| **신청 기준** | `REQUESTED` | 계약총액 − 선급금(신청) − 기성금누계(신청) | 신청된(청구된) 금액을 기준으로 계산 |
| **입금 기준** | `PAID` | 계약총액 − 선급금(입금) − 기성금누계(입금) | 실제 입금된 금액을 기준으로 계산 |

> **참고**: 잔금 탭의 "잔금 예정액" 은 서버에서 내려온 `balanceAmount`(잔여분)를 그대로 표시합니다. 잔여분이므로 실제 수금 여부와 무관하게 "아직 받아야 할 금액" 을 의미합니다.

### 잔금에 영향을 주는 선급금 차감

기성금 수금 시 미정산 선급금이 남아 있으면 **선급금 차감액** 이 발생하고, 잔금 단계에도 다음 항목이 함께 표시됩니다.

| 항목 | 의미 |
|------|------|
| 잔금 등록액 | 등록한 잔금 청구 금액 |
| 선급금 차감액 | 잔금에서 차감되는 미정산 선급금 (빨간색 `-` 표시) |
| 실수금액 | 잔금 등록액 − 선급금 차감액 (실제 입금 예정 금액) |

---

## 8.2 잔금 등록

잔금은 별도 "신청/승인" 절차 없이 **등록(생성) → 입금확인** 의 2단계로 처리됩니다.

> 📷 [스크린샷: 잔금 탭 - "잔금등록" 버튼이 활성화된 상태]

### 접근 경로
- 자금 상세 화면(`/admin/funds/{자금ID}`) → **잔금** 탭

### 등록 조건 (버튼 노출 규칙)

잔금 탭의 **[잔금등록]** 버튼은 다음 조건을 모두 만족할 때만 보입니다. (`FundBalanceTab` 기준)

1. **납품완료 상태**(`isDeliveryCompleted = true`) — 모든 출하의 납품확인이 끝나고 납품완료계가 완료되어야 함
2. 아직 **잔금 요청이 없음**(`hasBalanceRequest = false`)
3. 아직 **잔금 입금일이 없음**(`balancePaidDate` 없음)

> 납품완료 전이라면 잔금등록 버튼 대신 **[납품완료 처리하기]** 버튼이 표시됩니다. 먼저 납품완료 처리를 끝내야 잔금 등록이 가능합니다.

### 등록 절차

1. 잔금 탭에서 **[잔금등록]** 버튼 클릭
2. **잔금 등록 모달** 에서 입력
3. **[잔금 등록]** 버튼 클릭
4. 등록 완료 → 잔금 상태가 **"등록완료 (입금대기)"** 로 표시됨

> 모달 안내문: "잔금을 등록합니다. 등록 후 별도로 '잔금 입금확인'을 진행해야 입금 완료 처리됩니다."

### 입력 필드 (잔금 등록 모달)

| 필드 | 필수 | 설명 |
|------|------|------|
| 잔금 등록액 | 필수 | 등록할 잔금 금액 (자동 계산된 예정액이 초기값으로 채워짐, 수정 가능) |
| 등록일 | 필수 | 잔금 등록 날짜 (기본값: 오늘) |
| 비고 | 선택 | 메모 사항 |

### 버튼 / 동작

| 버튼 | 동작 |
|------|------|
| 잔금 등록 | 잔금 요청 레코드 생성 (등록액 > 0, 등록일 입력 시에만 활성화) |
| 취소 | 모달 닫기 |

## 8.3 잔금 입금확인 (수금 확인)

잔금을 등록한 뒤 실제로 돈이 입금되면 **잔금 입금확인** 을 진행해 수금 완료(PAID) 처리합니다.

> 📷 [스크린샷: 잔금 탭 하단의 "잔금 입금확인" 큰 초록 버튼 + 잔금 등록 정보]

### 절차

1. 잔금 탭 하단의 **[잔금 입금확인]** 버튼 클릭 (잔금 등록이 되어 있고 아직 입금완료(PAID)가 아닐 때만 표시)
2. **입금확인 모달**(공용 수금확인 모달, 제목 "잔금 입금확인")에서 입금 정보 입력
3. **[수금 확인]** 버튼 클릭
4. 완료 시 "잔금 입금 완료" 표시로 바뀜

> 잔금 등록 없이 입금확인을 시도하면 **"잔금 등록이 먼저 필요합니다."** 안내가 뜹니다.

### 입력 필드 (입금확인 모달)

| 필드 | 필수 | 설명 |
|------|------|------|
| 입금일 | 필수 | 실제 입금 날짜 (미래 날짜 선택 불가, 기본값 오늘) |
| 입금 금액 | 필수 | 실제 입금 금액. **[전액 입력]** 버튼으로 목표 금액 자동 입력 가능 |
| 입금 계좌 | 선택 | 예: 국민은행 123-456-789012 |
| 비고 | 선택 | 메모 |

> "남은 금액" 표시: 목표 금액(승인/실수금액) − 입력 금액 을 실시간으로 보여줍니다. 부분 입금도 가능합니다.

### 사용 예시 (잔금 계산)

| 항목 | 금액 |
|------|------|
| 계약총액(품대계) | 100,000,000원 |
| (−) 선급금 | 30,000,000원 |
| (−) 기성금 누계 | 50,000,000원 |
| **(=) 잔금 예정액** | **20,000,000원** |

만약 미정산 선급금이 2,000,000원 남아 잔금에서 차감된다면:

| 항목 | 금액 |
|------|------|
| 잔금 등록액 | 20,000,000원 |
| (−) 선급금 차감액 | 2,000,000원 |
| **(=) 실수금액 (입금 예정)** | **18,000,000원** |

### 오류 메시지 & 대응 (잔금)

| 메시지 | 발생 상황 | 원인 | 대응 |
|--------|-----------|------|------|
| 납품완료된 발주만 잔금 등록이 가능합니다. | 잔금 등록 시도 | 납품완료 전 상태 | 먼저 납품완료 처리 진행 |
| 납품완료 상태에서만 잔금 입금확인이 가능합니다. | 잔금 입금확인 시도 | 납품완료 전 상태 | 납품완료 처리 후 재시도 |
| 잔금 등록이 먼저 필요합니다. | 입금확인 버튼 클릭 | 잔금 미등록 | 먼저 [잔금등록] 진행 |
| 승인 상태에서만 수금 확인이 가능합니다. | 입금확인 시도 | 요청 상태가 수금확인 가능 단계가 아님 | 상태 확인 후 재시도, 지속 시 관리자 문의 |
| 입금일을 선택해주세요. / 입금 금액을 입력해주세요. | 모달 제출 | 필수 항목 누락 | 빈 필드 입력 |

## 8.4 OEM 지급 개요

OEM 지급은 **협력업체(제조사)에 지급** 하는 금액입니다. 일반적으로 **기성금의 70%** 를 지급합니다.

```
OEM 지급 예정액 = 기성금 누계 × 70%
```

> 화면 표기: OEM 지급 탭의 "OEM 지급 예정 (예정원가)" 은 `기성금누계 × 0.7` 로 자동 계산되어 표시됩니다.

### 중요 — 지급 관리 위치 변경 (읽기 전용 탭)

현재 자금 상세의 **OEM 지급 탭은 "읽기 전용"** 입니다. 실제 OEM 지급 등록/완료/취소는 **제조생산 > 월별 매출원장** 에서 관리합니다.

| 화면 | 역할 |
|------|------|
| 자금 상세 → OEM 지급 탭 | OEM 지급 현황 **조회**, 원가 재계산, B급 조정 |
| 제조생산 → 월별 매출원장 (`/admin/oem/monthly-ledger`) | OEM 지급 요청 → 확인 → 완료 **관리** |
| 제조생산 → OEM 대시보드 (`/admin/oem/dashboard`) | OEM 제조사별 발주/생산/재고/지급 통합 현황 |

> 📷 [스크린샷: 자금 상세 OEM 지급 탭 - 다크 요약 카드 + "월별 매출원장 바로가기" 안내 배너]

## 8.5 OEM 지급 탭 (조회 / 원가 재계산 / B급 조정)

### 접근 경로
- 자금 상세 화면 → **OEM 지급** 탭

### 화면 구성 (OEM 지급 요약 카드)

| 항목 | 설명 |
|------|------|
| 기성금 누계 | 청구된 기성금 합계 |
| OEM 지급 예정 (예정원가) | 기성금 누계 × 70% (또는 DB 저장 예정총액) |
| 지급 완료 | 실제 지급 완료(PAID)된 금액 합계 |
| 미지급 | 지급 예정 − 지급 완료 |

그 아래 **자금관리 기준 OEM 지급 이력** 표(읽기 전용)에 차수 / OEM 업체 / 지급예정금액 / 예정일 / 실제지급금액 / 지급일 / 상태가 표시됩니다.

| OEM 지급 상태 | 라벨 |
|---------------|------|
| `PENDING` | 지급예정 |
| `PAID` | 지급완료 |

### 버튼 / 동작

| 버튼 | 동작 | 비고 |
|------|------|------|
| 원가 재계산 | OEM 예정총액을 현재 원가 기준으로 재계산 | 원가 변동이 없으면 비활성("현재 원가와 동일합니다"), 납품완료 정보 없으면 비활성 |
| B급 조정 | B급(등외품) 품목 수량/금액 조정 | 출하 상태가 COMPLETED일 때만 가능 |
| 월별 매출원장 | `/admin/oem/monthly-ledger` 이동 | 실제 지급 관리 화면으로 이동 |

### 원가 재계산 사용 예시

OEM 원가가 변경되면 OEM 지급 탭 상단에 **노란 경고 배너** 가 뜹니다.

> "OEM 원가가 변경되어 재계산이 필요합니다 — 현재: 6,400,000원 → 재계산 시: 6,800,000원 (+400,000원)"

1. 배너 또는 헤더의 **[원가 재계산]** 버튼 클릭
2. 확인 창: "OEM 예정총액을 현재 원가 기준으로 재계산합니다." → 확인
3. 완료 시 "재계산이 완료되었습니다." 안내

> 재계산은 먼저 **미리보기**(DB 변경 없음)로 차이를 보여준 뒤 실행됩니다. 실행 전에는 DB 값이 바뀌지 않습니다.

### 오류 메시지 & 대응 (OEM 지급 탭)

| 메시지 | 발생 상황 | 원인 | 대응 |
|--------|-----------|------|------|
| 재계산에 실패했습니다. | 원가 재계산 실행 | 서버 처리 오류 | 잠시 후 재시도, 지속 시 관리자 문의 |
| 존재하지 않는 자금 관리 정보입니다. | OEM 조회/재계산 | 자금 정보 없음/삭제됨 | 목록 새로고침 후 재진입 |
| 존재하지 않는 OEM 지급 이력입니다. | 지급 처리 | 대상 지급건 없음/삭제됨 | 화면 새로고침 |
| 이미 지급 완료된 내역입니다. | 지급 완료 재시도 | 이미 PAID 상태 | 새로고침 후 상태 확인 |

## 8.6 OEM 지급 관리 (월별 매출원장)

실제 OEM 지급 요청·확인·완료는 **제조생산 > 월별 매출원장** 에서 처리합니다.

| 구분 | 내용 |
|------|------|
| **메뉴 경로** | 제조생산 → 월별 매출원장 |
| **라우트** | `/admin/oem/monthly-ledger` |

> 📷 [스크린샷: 월별 매출원장 - OEM 제조사 선택 + 연월 선택 + 원장 표 + 지급요청/확인/완료 버튼]

### 절차

1. **OEM 제조사** 선택 (OEM 담당자 계정은 자사로 자동 고정·잠금)

3. 해당 월 매출원장 합계를 기준으로 **[지급 요청]**

# 9. FAQ 및 문제해결

## 9.1 자주 묻는 질문

### 발주 / 계약 관련

**Q: PDF 업로드 후 정보가 정확하게 추출되지 않습니다.**
> A: AI/OCR 추출은 PDF 품질에 따라 정확도가 달라질 수 있습니다. 추출 후 반드시 정보를 확인하고 필요 시 수동으로 수정하세요. 텍스트가 없는 스캔 PDF는 추출 정확도가 떨어집니다.

**Q: "PDF 처리 완전 실패: 어떠한 데이터도 추출할 수 없습니다." 가 뜹니다.**
> A: 손상되었거나 텍스트가 전혀 없는 PDF일 수 있습니다. 다른 PDF로 다시 시도하거나, 발주 정보를 수동 입력하세요. (참고: PDF만 업로드 가능, 파일 크기 100MB 이하)

**Q: 변경계약을 등록했는데 본계약과 연결되지 않습니다.**
> A: 변경계약 등록 시 **상위 계약** 필드에서 반드시 본계약을 선택해야 합니다. 선택하지 않으면 별도의 본계약으로 등록됩니다.

**Q: 완료된 발주를 수정하려는데 안 됩니다.**
> A: 발주가 **완료(COMPLETED)** 상태가 되면 수정·삭제·출하등록이 모두 제한되며 조회·PDF 다운로드만 가능합니다.

### 출하 관련

**Q: 출하 수량을 변경하고 싶은데 변경 버튼이 비활성화되어 있습니다.**
> A: 해당 출하가 이미 **기성금 청구에 포함**되었거나 **납품확인 완료(COMPLETED)** 상태이면 수량 변경이 불가능합니다. 수량 조정(추가변경)은 출하가 **진행중(IN_PROGRESS)** 일 때만 가능합니다.

**Q: 출하를 삭제하려는데 "완료된 출하는 취소할 수 없습니다." 가 뜹니다.**
> A: 납품확인이 완료된 출하는 취소·삭제할 수 없습니다. 또한 출고요청(dispatch)이 걸려 있는 출하도 제약이 있습니다.

**Q: 출하 등록 시 "OEM 제조사를 선택해주세요." 가 뜹니다.**
> A: OEM 생산 품목 출하는 제조사 선택이 필수입니다. 제조사를 지정한 뒤 다시 등록하세요. 단, **출고요청이 진행 중인 출하는 OEM 제조사를 변경할 수 없습니다.**

**Q: "출하수량이 발주 잔여수량을 초과" 합니다.**
> A: 출하 수량 합계는 발주 수량을 넘을 수 없습니다(발주수량 ≥ Σ출하수량). 잔여수량 범위 내에서 출하하세요.

**Q: "수요기관번호가 5자리 미만입니다" 로 출하번호 생성이 안 됩니다.**
> A: 출하번호는 수요기관번호를 기반으로 생성됩니다. 발주의 수요기관번호가 올바르게 입력되어 있는지 확인하세요.

### 납품확인 (모바일) 관련

**Q: 모바일에서 납품확인 URL에 접속이 안 됩니다 / "유효하지 않은 접근 링크입니다."**
> A: URL 유효기간(24시간)이 지났거나 토큰이 만료되었을 수 있습니다. 관리자에게 요청해 새 URL을 발송받으세요.

**Q: 납품 완료가 안 되고 "현장소장 서명이 필요합니다." / "납품 사진이 필요합니다." 가 뜹니다.**
> A: 납품확인 완료에는 **인수자(현장소장) 서명 + 사진 최소 1장** 이 모두 필요합니다. 서명과 사진을 등록한 뒤 완료 처리하세요.

**Q: 사진을 더 올리고 싶은데 "사진은 최대 5장까지 업로드 가능합니다." 가 뜹니다.**
> A: 출하(납품)당 사진은 최대 5장입니다. 기존 사진을 교체하거나 일부를 삭제한 뒤 추가하세요.

**Q: 서명/사진을 수정하려는데 "이미 완료된 납품에는 사진을 추가할 수 없습니다." 가 뜹니다.**
> A: 납품확인이 완료된 건은 수정이 제한됩니다. 관리자에게 문의해 처리 방법을 확인하세요.

**Q: 문자(서명 URL)가 안 갑니다 / "인수자 연락처가 등록되지 않았습니다."**
> A: 운송장에 인수자 연락처가 등록되어 있어야 서명 URL을 발송할 수 있습니다. 운송 정보에서 인수자 연락처를 먼저 입력하세요.

### 기성금 / 기성차수 관련

**Q: 기성금 청구 시 일부 출하가 목록에 표시되지 않습니다.**
> A: 다음을 확인하세요. ① 해당 출하의 납품확인(인수증)이 완료되었는지 ② 이미 다른 기성 청구에 포함되었는지. (인수증 미완료/이미 청구된 출하는 선택할 수 없습니다.)

**Q: "이미 기성 청구된 출하입니다." / "인수증이 완료되지 않은 출하가 포함되어 있습니다." 가 뜹니다.**
> A: 기성 차수에는 **납품확인(인수증) 완료된, 아직 청구되지 않은 출하** 만 포함할 수 있습니다.

**Q: 기성 차수를 삭제할 수 없습니다.**
> A: 기성 차수는 **마지막 차수만 삭제 가능** 하며, **납품확인서가 연결된 차수는 삭제할 수 없습니다.**

**Q: "납품완료된 발주는 기성금 청구가 불가능합니다. 잔금 등록을 이용하세요." 가 뜹니다.**
> A: 납품완료 처리된 발주는 더 이상 기성금 청구를 할 수 없습니다. 남은 금액은 **잔금 등록** 으로 정산하세요.

### 자금 (선급금 / 잔금 / OEM) 관련

**Q: 선급금을 다시 신청하고 싶습니다.**
> A: 선급금은 **1회만 신청 가능** 합니다. 금액 변경이 필요하면 요청 상태에서 수정하거나 관리자에게 문의하세요.

**Q: "기성청구가 등록되어 있어 선급금 신청이 불가능합니다." 가 뜹니다.**
> A: 이미 기성청구가 진행된 발주는 선급금을 새로 신청할 수 없습니다. 선급금은 통상 계약 초기(첫 출하 전후)에 신청합니다.

**Q: 선급금을 삭제할 수 없습니다 / "입금확인이 완료된 선급금은 삭제할 수 없습니다."**
> A: **입금확인(PAID) 전** 상태의 선급금만 삭제할 수 있습니다. 삭제 시 관련 PDF도 함께 삭제됩니다.

**Q: 잔금 등록 버튼이 보이지 않습니다.**
> A: 잔금등록 버튼은 **납품완료 상태 + 잔금 미등록 + 잔금 미입금** 일 때만 보입니다. 납품완료 전이면 먼저 **[납품완료 처리하기]** 를 진행하세요.

**Q: OEM 지급을 자금 탭에서 등록하려는데 버튼이 없습니다.**
> A: OEM 지급 등록·완료는 **제조생산 > 월별 매출원장**(`/admin/oem/monthly-ledger`)에서 관리합니다. 자금 상세의 OEM 탭은 **조회 전용**(원가 재계산·B급 조정만 가능)입니다.

**Q: OEM 지급 예정액은 어떻게 계산되나요?**
> A: 기본적으로 **기성금 누계 × 70%** 입니다. 원가가 변동되면 OEM 탭에서 **[원가 재계산]** 으로 예정총액을 현재 원가에 맞게 갱신할 수 있습니다.

**Q: 수금률이 100%가 안 되는데 자금 상태가 완료로 안 바뀝니다.**
> A: 자금은 **선급금·기성금·잔금이 모두 입금확인(PAID)** 되어야 완료(COMPLETED)로 전환됩니다. 미입금 항목이 있는지 확인하세요.

### 권한 / 로그인 관련

**Q: "권한이 없습니다 / 접근 권한이 없습니다." 가 뜹니다.**
> A: 해당 기능에 대한 권한이 없는 계정입니다. 자금·기성 관리는 시스템관리자·리드파워담당자만 접근할 수 있습니다. 관리자에게 권한을 요청하세요.

**Q: 로그인 직후 일부 화면(메뉴/알림)이 잠깐 비거나 새로고침하면 정상입니다.**
> A: 로그인 직후 일부 초기 요청이 인증 완료 직전에 먼저 나가면서 일시적으로 비어 보일 수 있습니다. 새로고침하면 정상 동작합니다. (자가 복구되는 현상으로 기능에는 지장 없음) *(확인 필요: 향후 인증 부트스트랩 순서 보완 예정)*

## 9.3 문제해결 시나리오 ("이럴 땐 이렇게")

### 시나리오 1 — 잔금 등록 버튼이 안 보여요

1. 잔금 탭의 **납품완료 상태** 가 "완료" 인지 확인합니다.
2. "미완료" 라면 먼저 **[납품완료 처리하기]** 버튼으로 납품완료를 진행합니다. (모든 출하 납품확인 완료 + 양측 서명 필요)
3. 납품완료 후에도 안 보이면 이미 잔금이 등록되었거나 입금완료된 상태입니다. 잔금 등록 상태/입금일 표시를 확인하세요.

### 시나리오 2 — 잔금 금액이 예상과 달라요

1. 상단 요약 카드에서 **계약총액 / 선급금 / 기성금 누계** 를 확인합니다.
2. `잔금 = 계약총액 − 선급금 − 기성금누계` 로 직접 검산합니다.
3. **선급금 차감액** 이 표시되면, 미정산 선급금이 잔금에서 차감되어 **실수금액** 이 줄어든 것입니다(정상).
4. 그래도 다르면 기성금 청구 누계가 최신인지, 변경계약으로 계약총액이 바뀌었는지 확인합니다.

### 시나리오 3 — OEM 예정총액이 실제 원가와 안 맞아요

1. 자금 상세 → **OEM 지급** 탭으로 이동합니다.
2. 상단에 노란 **재계산 경고 배너**("OEM 원가가 변경되어 재계산이 필요합니다")가 있는지 확인합니다.
3. 배너의 **[원가 재계산]** 으로 변경 전/후 금액을 확인하고 실행합니다.
4. 실제 지급 관리는 **월별 매출원장** 에서 진행합니다.

### 시나리오 4 — 모바일 납품확인 링크가 만료됐어요

1. 사용자에게 "유효하지 않은 접근 링크" 메시지가 보이는지 확인합니다.
2. 관리자 화면(운송/납품확인)에서 해당 건의 **서명 URL을 재발송** 합니다. (유효기간 24시간)
3. 인수자 연락처가 비어 있으면 운송장에 연락처를 먼저 입력해야 발송됩니다.

### 시나리오 5 — 기성금 청구 목록에 출하가 안 나와요

1. 그 출하의 **납품확인(인수증)이 완료** 되었는지 확인합니다(미완료면 청구 불가).
2. 이미 **다른 기성 차수에 포함** 되었는지 확인합니다(중복 청구 불가).
3. 발주가 이미 **납품완료** 상태면 기성금 대신 **잔금 등록** 으로 정산합니다.

### 시나리오 6 — 화면이 비어 보이거나 권한 오류가 나요

1. 먼저 **새로고침(F5)** 합니다. (로그인 직후 일시적 현상일 수 있음)
2. "접근 권한이 없습니다" 가 계속되면 본인 계정 권한을 관리자에게 확인합니다.
3. "세션이 만료되었습니다" 면 **다시 로그인** 합니다.
